# Pipe Pay Theme Review - 2026-05-07

> Source: `superpowers:code-reviewer` agent. Scope: `pipepay-child/` (all PHP + style.css). Focus: escaping, WC hooks, hardcoded URLs, accessibility, SEO, mobile, performance, CSS quality, template architecture.

## Critical
None. No XSS exposure, no leftover `pipepay.money`, no leaked secrets, no broken WC dequeue logic.

## High

**`page-contact.php:26` - unescaped echo into an `href` attribute**
```php
href="mailto:...?subject=<?php echo $mail_subject; ?>"
```
`$mail_subject` is `rawurlencode( 'Pipe Pay - ' )` so today the value is safe, but the pattern is wrong: any future change that puts dynamic data into `$mail_subject` becomes an attribute-injection vector. Fix: `echo esc_attr( $mail_subject )`. The `// already URL-encoded` comment is misleading - URL-encoding does not satisfy attribute escaping.

**`functions.php:140-171` - mobile drawer accessibility gaps**
- No `inert`/focus-trap on the open drawer; Tab walks straight off the panel into hidden header/body content.
- ESC closes the drawer but focus is not returned to `.pp-nav-toggle` (screen readers + keyboard users lose their place).
- Hamburger button is 40×40 (`style.css:423-424`); WCAG 2.5.5 AAA recommends 44×44 and Apple HIG requires 44pt. Bump to 44×44.
- Visible nav `<a>` items inside the drawer have no visible focus styles overriding the default browser ring on white background.

**No skip-to-content link anywhere** (`header.php`, `front-page.php`). WCAG 2.4.1 fail. Add a visually-hidden-until-focused `<a class="pp-skip" href="#content">Skip to content</a>` as the first child of `<body>`, and put `id="content"` on `<main>`.

**`functions.php:67-76` - SEO meta only on the homepage**
`og:*`, `twitter:card`, and `<meta name="description">` are emitted only when `is_front_page()`. `/how-it-works`, `/pricing`, `/docs/*` get a `<title>` but no description, no OG, no canonical, no `og:image`, no `twitter:image`. Twitter `summary_large_image` without an image URL is broken. Add per-template description + canonical + a default `og:image`.

**No `<link rel="canonical">` anywhere.** WordPress emits `rel_canonical()` on singular pages by default, but the homepage and the front-page meta block don't include one. Add one in the same `wp_head` hook.

## Medium

**`footer.php:55` and `front-page.php:511` - hardcoded version "v1.6.2"**
Duplicated string. CLAUDE.md notes the plugin is now 1.6.3, so the footer is already stale. Move to a constant (e.g. `PIPEPAY_SITE_VERSION` defined in `functions.php`) and echo via `esc_html()`.

**`front-page.php:484-514` - footer markup duplicated from `footer.php`**
The home page inlines its own copy of the footer instead of `require __DIR__ . '/footer-content.php'`. Two copies will drift (already drifting: footer links on home use absolute `https://pipepay.app/...` strings, whereas `footer.php` uses `home_url()`). Same problem with the four inline logo SVGs: 3 identical copies + 1 inverse. Extract to a `partials/logo-svg.php` (or a `pp_logo_svg($variant)` helper).

**`front-page.php:19-24` - hardcoded absolute URLs**
```php
$docs_url = 'https://pipepay.app/docs';
```
Use `home_url('/docs')` (already used everywhere else). Breaks in a staging or local dev environment.

**`functions.php:67` - `is_front_page()` check inside `wp_head` is fine, but the `og:url` is hardcoded** (`'https://pipepay.app/'`). Use `esc_url( home_url( '/' ) )`.

**`functions.php:51` - `add_theme_support('title-tag')` is registered inside `after_setup_theme` correctly, but the comment ("Allow the homepage template to skip GeneratePress's default markup wrappers") is unrelated to what the line does. Misleading.**

**`functions.php:178-187` - `woocommerce_add_cart_item_data` filter empties cart unconditionally**
If a customer has a non-Pipe-Pay product (theoretically possible if you ever add merch) and clicks a Pipe Pay tier, their cart vanishes silently. Today there are only Pipe Pay products in the catalog so this is benign - flag it for the day that changes.

**`style.css:2884-2885` - two `!important` rules on `.sidebar.widget-area` and `.content-area`**
GeneratePress filter `generate_sidebar_layout` (functions.php:191) already returns `'no-sidebar'` for WC pages, which should make this CSS unnecessary. Audit whether the filter actually wins; if so, drop the `!important` overrides.

**`style.css` - 45 `!important` declarations across 3,618 lines**
Concentrated mostly on `text-decoration: none !important` on link styles and a few WC overrides. Most can be removed by writing more specific selectors instead.

## Low

**`footer.php:51`, `front-page.php:507` - `date('Y')` instead of `wp_date('Y')`**
Uses server's PHP timezone instead of WP's site-configured timezone. Edge case (year boundary at midnight) but trivial to fix.

**`functions.php:140-171` - inline footer JS** uses `var` and IIFE patterns that work everywhere; minor nit.

**`style.css:1473, 1483, 1488` - `!important` on `.pp-shiplog__more` color** is a specificity war with GeneratePress link defaults.

**Performance** - `style.css` is 3,618 lines and unminified, served as a separate file. At ~108 KB raw it's fine over Cloudflare with Brotli (~20 KB on wire). Consider splitting WC-specific overrides (lines 2876–end) into a separate handle that only enqueues on WC pages - pairs nicely with the existing dequeue work.

**Mobile responsiveness** - the hamburger breakpoint is 760px (`style.css:2831`); 768px iPad portrait shows the desktop nav. Intentional? If iPad portrait should show the drawer, bump to `max-width: 800px`.

**`<form class="pp-footer-signup">` POSTs to `/contact`** without CSRF or capture. There's no handler for it server-side - it just navigates to the contact page. Either wire it to a real subscribe endpoint or change the form to a simple `<a href="/contact">` link.

## Overall Theme Quality
Solid hand-coded WordPress child theme. Escaping discipline is good: every `esc_url`/`esc_html`/`wp_kses_post` call is in the right place, and the four `phpcs:ignore` annotations are correctly used for static, author-controlled markup. WC hook integration is clean and idiomatic - the dequeue-on-non-WC-pages pattern (priority 99) is exactly the right shape, the `generate_sidebar_layout` filter and `woocommerce_before/after_main_content` hooks are wired with sensible priorities, and the cart-dedup `woocommerce_add_cart_item_data` filter is appropriately scoped to the four product IDs. The real weak spots are accessibility (no skip link, drawer focus management absent, 40px touch targets) and SEO completeness on non-home templates (missing canonical/description/OG on /how-it-works, /pricing, /docs/*). Code duplication across templates is the next maintenance debt to pay down - four logo SVGs and two footer markup copies will silently drift. None of this is shipping-blocking; the theme is launch-ready and a couple of focused passes (skip link + per-template SEO meta + extract logo partial + 44px hamburger) would put it in genuinely good shape.
